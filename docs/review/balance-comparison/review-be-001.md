# BE Code Review — balance-comparison · Iteration 1

> **Reviewer:** `banking-reviewer-be` (Principal Backend Engineer)
> **Sprint:** SPRINT-2026-Q2-BC-01
> **Date:** 2026-05-21
> **Artifact under review:** `backend/balance-dashboard-service/` + `common-libs/account-client-lib/` + `backend/account-service/src/main/resources/db/migration/V002__add_accounts_customer_id_idx.sql`
> **Reference:** TL-001 (task-plan, implementation-notes, ADR-006, ADR-007)

## Verdict: changes_requested

**Rationale:** 2 blockers — (1) the staging profile explicitly disables Redis TLS, directly violating Security C-4 / Security hard rule (TLS in transit for cache); (2) the hexagonal-architecture purity rule is broken because `application/` imports an exception type from `infrastructure/`, contradicting task-plan §Architecture Constraints and the explicit Layer 2 contract for `AccountPort`. Several major issues around test coverage gaps for controller layer (no `@WebMvcTest` for `BalanceDashboardController` even though 11 cases are itemized in task-plan §Layer 5), Jackson serialization of `BigDecimal balance` (annotation on private field instead of getter — silent risk of number-not-string output), and dead `BalanceSnapshot` record.
