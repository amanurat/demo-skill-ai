# Kafka Anti-Patterns — Banking Reviewer Catalog

Each entry: **what / why bad / severity / detection / fix**.

Severity legend: `blocker` = auto-fail review · `major` = must fix before merge · `minor` = log as comment.

---

## 1. `kafkaTemplate.send()` inside business `@Transactional` — **blocker**

**What.** Publishing to Kafka from the same method that writes to DB.

**Why bad.** Dual-write hazard. If DB commit fails after send → consumer sees event that "never happened". If send fails after DB commit → state diverges silently.

**Detect.** Grep `kafkaTemplate.send` inside any class annotated `@Service` whose methods are `@Transactional`. Also check listener methods that mutate DB and emit events.

**Fix.** Use Outbox. See [`outbox-dispatcher.md`](outbox-dispatcher.md).

---

## 2. `enable.auto.commit=true` on consumer — **blocker**

**What.** Default consumer config that auto-commits offsets every 5s.

**Why bad.** Offsets commit before processing finishes → crash mid-batch loses messages.

**Detect.** Check `application.yml` and `ConsumerFactory` config for `enable.auto.commit`.

**Fix.** `enable.auto.commit=false`, set `AckMode.MANUAL_IMMEDIATE` or `BATCH`, ack only after success.

---

## 3. No DLT on financial consumer — **blocker**

**What.** Listener has no `DeadLetterPublishingRecoverer` or `@RetryableTopic`.

**Why bad.** Poison message infinitely retries → consumer falls behind → entire group blocked.

**Detect.** Every `@KafkaListener` factory should reference an error handler with DLT routing.

**Fix.** Wire `DefaultErrorHandler` + `DeadLetterPublishingRecoverer` in container factory.

---

## 4. Producer not idempotent — **blocker**

**What.** `enable.idempotence=false` or unset (defaults vary by client version).

**Why bad.** Retry on network blip → duplicate events → double-debits downstream.

**Detect.** Producer factory missing `enable.idempotence=true` or has `max.in.flight > 5`.

**Fix.** Set `enable.idempotence=true`, `acks=all`, `max.in.flight.requests.per.connection ≤ 5`.

---

## 5. Null partition key when ordering matters — **major**

**What.** Sending records with `null` key for a topic where per-aggregate order matters.

**Why bad.** Round-robin partitioning → events for the same account land on different partitions → out-of-order processing.

**Detect.** Producer code passing `null` key for financial topics; topic has > 1 partition.

**Fix.** Use the aggregate ID (e.g., `accountId`, `transferId`) as key.

---

## 6. Async send without callback or error metric — **major**

**What.** `kafkaTemplate.send(record);` with no `.whenComplete()` or `ProducerListener`.

**Why bad.** Send failures are invisible — no alert, no retry.

**Detect.** Grep for `kafkaTemplate.send(` not followed by `.get(` or `.whenComplete(` and no global `setProducerListener`.

**Fix.** Either sync `.get()` (Outbox path) or callback that logs + increments error counter.

---

## 7. Consumer not idempotent — **major**

**What.** Handler that mutates state without checking `event-id`.

**Why bad.** At-least-once delivery + retry → duplicate side-effects (double-credit, double-notification).

**Detect.** Listener calls repository `save`/`update` with no inbox check or natural unique constraint.

**Fix.** Add `inbox` table on `event-id` PK, check before processing. Or rely on natural idempotency (e.g. `INSERT … ON CONFLICT DO NOTHING`).

---

## 8. Logging full payload at INFO/DEBUG — **major**

**What.** `log.info("received {}", event);` where event contains PII or money.

**Why bad.** Violates banking-hard-rule "No PII in logs". Triggers compliance finding.

**Detect.** Grep `log.(info|debug)\(".*\{\}.*", .*event` in listeners; check `toString()` generated by Lombok `@Data` on event records.

**Fix.** Log identifiers only (`eventId`, `aggregateId`). Use redacted DTO if payload must be inspected.

---

## 9. JSON `String` payload for financial events — **major**

**What.** Producer sends `kafkaTemplate.send(topic, key, jsonString)` with no schema.

**Why bad.** No contract enforcement — breaking changes ship silently, consumers crash in prod.

**Detect.** `StringSerializer` configured as value serializer for financial topics.

**Fix.** Use `KafkaAvroSerializer` (or Protobuf) + Schema Registry. See [`schema-registry-avro.md`](schema-registry-avro.md).

---

## 10. `auto.register.schemas=true` in production — **major**

**What.** Producer registers new schema versions at runtime instead of CI.

**Why bad.** Breaking schema ships unreviewed; rollback is hard once registry has it.

**Detect.** Search producer config for `auto.register.schemas` not set to `false`.

**Fix.** `auto.register.schemas=false` in prod; register via maven plugin in CI with `BACKWARD_TRANSITIVE` check.

---

## 11. Long-running processing > `max.poll.interval.ms` — **major**

**What.** Listener does HTTP call that takes 5+ minutes; default `max.poll.interval.ms=300000`.

**Why bad.** Consumer kicked from group, partitions rebalanced, processing retried → duplicate delivery, lag spike.

**Detect.** Listener calls slow external service synchronously without bumping `max.poll.interval.ms` or offloading to async worker.

**Fix.** Either bump `max.poll.interval.ms`, reduce `max.poll.records`, or move work to async worker + small ack window.

---

## 12. Single dispatcher pod with no `SKIP LOCKED` — **minor**

**What.** Outbox dispatcher uses ShedLock or `@Scheduled` without distributed lock; throughput tied to one pod.

**Why bad.** Slow burst processing; pod restart pauses all delivery.

**Detect.** Outbox repo missing `FOR UPDATE SKIP LOCKED` and no ShedLock annotation either.

**Fix.** Use `FOR UPDATE SKIP LOCKED` and let multiple pods cooperate.

---

## 13. Concurrency > partitions — **minor**

**What.** `factory.setConcurrency(10)` on a topic with 6 partitions.

**Why bad.** Extra threads idle; wastes memory and confuses ops.

**Detect.** Compare configured concurrency × pod count vs partition count.

**Fix.** `concurrency ≤ partitions / pods`. Adjust when partitions are scaled.

---

## 14. No metric on consumer lag — **minor**

**What.** Service has Kafka listeners but Micrometer Kafka binding not enabled.

**Why bad.** Lag spikes invisible — first signal is angry stakeholder.

**Detect.** Check actuator endpoint `/actuator/metrics` for `kafka.consumer.records-lag-max`. Absent if binding missing.

**Fix.** `KafkaClientMetrics` registered as bean (auto-config covers this in Spring Boot 3.x — verify dep on `micrometer-core`).

---

## 15. Topic created at runtime by `KafkaAdmin` — **minor**

**What.** Service declares `@Bean NewTopic` and Spring auto-creates on startup.

**Why bad.** Production topics live longer than apps. Partitions, retention, replication should be infra-as-code (Terraform), reviewed by ops.

**Detect.** Grep `NewTopic` beans in production profiles.

**Fix.** Move topic provisioning to Terraform / Pulumi. Auto-create only in test profile.
